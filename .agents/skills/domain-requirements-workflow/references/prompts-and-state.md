# Prompts And State

## Purpose

Use this reference when implementing the skill as a multi-role workflow or when preparing the working files for a requirements-discovery run.

## Recommended Working Files

Maintain these files throughout the run:

1. `state.json`
2. `source-ledger.json`
3. `evidence-log.json`
4. `final-requirements.md`

Claim-to-evidence traceability is not a separate file. Need and requirement records inside `state.json` carry their own `supported_by` (and `need_ids`) links; the final document renders the mapping from them.

## Role Prompts

Run the Critic and the stage 12 validity review in a fresh context — a separate subagent or session that sees the artifacts but not the reasoning that produced them. The author scoring its own work is the weakest check in this workflow; a fresh reviewer is what makes self-scoring meaningful. The Orchestrator, Researcher, and Editor can share the main context.

### Orchestrator

```text
Act as the workflow orchestrator.
Keep the run aligned to the decision question.
Do not allow stage transitions until completion checks are met.
Refuse to accept major needs or requirements with no supporting evidence.
Keep gaps, conflicts, and undecided items visible.

Output:
- current stage
- pass or fail
- next role
- return reason if any
- state fields to update
```

### Researcher

```text
Act as the researcher.
Read materials, conduct structured interviews, update the domain map, and extract evidence.
Extract evidence only from real sources: real documents, real stakeholder conversations, observation.
You may gather real sources by web/literature search instead of waiting for the user, but web is general-domain only and a hypothesis unless from an authoritative named source; record the query and access date, and never let it stand in for org-specific operations or a confirmed requirement.
Verify each web URL resolves (fetch it) before logging it, and record the result in url_verified; never cite a URL you have not confirmed exists.
Simulated persona dry-runs yield hypotheses and questions, never evidence — never give them a source_id.
Register every source at first use.
Record observations rather than conclusions whenever possible.
Classify each evidence fragment as fact, inference, or request.

Output:
- new sources
- new evidence fragments
- domain updates
- new unknowns
- new conflicts
```

### Critic

```text
Act as the critic.
Inspect the run for unsupported claims, weak evidence, terminology drift, and conflicts.
Score sufficiency and say whether the run can proceed.

Output:
- unsupported claims
- weak evidence areas
- conflicts
- re-exploration targets
```

### Editor

```text
Act as the editor.
Shape needs and requirements while preserving traceability.
Separate confirmed items from undecided items.
Keep the final document concise and auditable.
In the source list, show each web source's verified URL (say so plainly when none was captured); render the evidence fragments and the claim-to-evidence mapping as markdown tables.

Output:
- need list
- requirement list
- undecided list
- claim-to-evidence mapping
```

## Need Record Shape

```json
{
  "need_id": "N-001",
  "statement": "Reduce delay in exception handling",
  "stakeholders": [
    "operations staff"
  ],
  "background": "Exception approvals create visible waiting time.",
  "priority": "high",
  "confidence": "medium",
  "status": "supported",
  "supported_by": [
    "E-001",
    "E-007"
  ]
}
```

`status` is `draft` until real-source evidence supports the claim, then `supported`. The claim-to-evidence mapping in the final document is generated by walking these records — there is no separate traceability file to keep in sync.
